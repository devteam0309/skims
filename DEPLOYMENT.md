# SKIMS — Deployment Guide (Netlify + Render)

- **Frontend** (Vite/React) → Netlify (static)
- **Backend** (Express API) → Render (Node web service)
- **Database** → MongoDB Atlas
- **File storage** → Cloudinary (uploads never touch the server disk)

The frontend calls the backend through a relative `/api` path. Netlify **proxies** `/api/*`
to Render (see `netlify.toml`), so the browser stays same-origin with Netlify — this is what
keeps the `SameSite=strict` auth cookies working. Do not point the frontend directly at the
Render domain unless you also change the cookies to `SameSite=none`.

---

## Order of operations

1. **MongoDB Atlas** — create a cluster, a DB user, and set Network Access to allow `0.0.0.0/0`
   (Render free tier has no static outbound IP). Copy the connection string → `MONGO_URI`.
2. **Cloudinary** — grab cloud name / API key / API secret.
3. **SMTP** — e.g. a Gmail account with an App Password.
4. **Render** — deploy the backend (Blueprint from `render.yaml`), fill in the env vars below.
   Note the service URL, e.g. `https://skims-api.onrender.com`.
5. **Edit `netlify.toml`** — replace the proxy host with your real Render URL, commit, push.
6. **Netlify** — deploy the frontend (it auto-detects `netlify.toml`). Note the site URL.
7. **Back on Render** — set `CLIENT_URL` to the Netlify URL and redeploy.
8. **Seed** the database once: `npm run seed` (from `backend/`, with prod env) or via Render Shell.

## Render — backend env variables

| Variable | Value / Notes |
|---|---|
| `NODE_ENV` | `production` *(set in render.yaml)* |
| `PORT` | **Do not set** — Render injects it automatically. |
| `MONGO_URI` | Atlas connection string (include the DB name). |
| `JWT_SECRET` | Auto-generated by Render, or `node -e "console.log(require('crypto').randomBytes(64).toString('hex'))"`. |
| `JWT_EXPIRE` | `15m` *(set in render.yaml)* |
| `CLIENT_URL` | Your Netlify site URL, e.g. `https://skims.netlify.app` (no trailing slash). |
| `CLOUDINARY_CLOUD_NAME` | from Cloudinary |
| `CLOUDINARY_API_KEY` | from Cloudinary |
| `CLOUDINARY_API_SECRET` | from Cloudinary |
| `SMTP_HOST` | e.g. `smtp.gmail.com` |
| `SMTP_PORT` | e.g. `587` |
| `SMTP_EMAIL` | sending account address |
| `SMTP_PASSWORD` | SMTP / app password |
| `FROM_EMAIL` | e.g. `noreply@skims.gov.ph` |
| `FROM_NAME` | `SKIMS Platform` *(set in render.yaml)* |
| `MAX_FILE_SIZE` | `10485760` (10 MB) *(set in render.yaml)* |

## Netlify — frontend env variables

**None required**, and that is the secure default. The app uses a relative `/api` base and the
proxy in `netlify.toml`; `NODE_VERSION=20` is pinned there.

> ⚠️ **Do NOT set `VITE_SHOW_QA_CREDS`** on the public Netlify site. This flag (used only for
> dev/QA builds) compiles the seeded test-account credentials *and* the internal `/ref` system
> reference page into the bundle. With the flag absent, both are stripped from the production
> build at compile time (verified: the strings are not present in `dist`).

## Seeding the production database

The seeder **deletes all collections** before inserting, so it is guarded:

- In production it refuses to run unless `SEED_CONFIRM=true`.
- Demo accounts (the shared `Admin@123` logins) are **opt-in**: they are only created when
  `SEED_DEMO=true`. By default a production seed inserts reference data (municipalities +
  barangays) and — if `SEED_ADMIN_EMAIL` / `SEED_ADMIN_PASSWORD` are set — a single real
  super-admin.

Recommended first prod seed (reference data + your own admin, no demo logins):

```
NODE_ENV=production SEED_CONFIRM=true \
SEED_ADMIN_EMAIL=admin@yourdomain.gov.ph SEED_ADMIN_PASSWORD='<strong-unique>' \
npm run seed
```

## Gotchas

- **Free-tier sleep:** Render free services sleep after ~15 min idle (~50s cold start). The
  in-process `node-cron` jobs (daily deadline reminders, 6-hourly compliance checks) will **not
  run while asleep**. Use a paid instance or an external scheduler hitting an endpoint if these matter.
- **Atlas IP allowlist:** must include `0.0.0.0/0` for Render free tier.
- **CLIENT_URL** must exactly match the Netlify origin — it drives both CORS and the CSP `connectSrc`
  and is used to build links in outgoing emails.
- **Proxy host:** keep the Render URL in `netlify.toml` in sync with your actual backend.
